# Slice: TUI Phase 3 тАФ live model picker, session switcher, usage meter

Date: 2026-06-15
Roadmap: `docs/initiatives/tui.md` Phase 3 (Polish + remaining cloud).
Priority: round out the interactive agent TUI with the three remaining Phase 3
polish items that do not require new backend work.

## Goal

Make the `codel00p` agent TUI feel finished for day-to-day use:

1. **Live model picker** тАФ F2 / `/model` opens a picker populated from the
   provider's live `list_models` catalog, not just a hand-maintained list.
2. **Session switcher** тАФ F5 / `/switch` lists prior conversations and resumes the
   selected one in place, without leaving the TUI.
3. **Token/usage meter** тАФ the status bar surfaces a running message count and an
   estimated token total for the current conversation.

## Design

Stays inside the Elm architecture already used by the TUI: state changes only via
`Msg`/`update`; side effects only via `Effect`; the event loop is the sole impure
driver. New work is confined to `crates/codel00p-cli/src/tui/` plus two tiny
helpers in the agent/providers modules (existing public APIs only).

### Model picker (`Effect::FetchModels` тЖТ `Msg::Models`)

- `Overlay::Model` now holds a `ModelPicker { picker: Picker<ModelChoice>, status }`
  so a transient "Loading modelsтАж" / fallback line can sit above the rows.
- Opening it shows the **static catalog immediately** (usable offline) and fires
  `Effect::FetchModels(provider)`. The event loop runs `list_provider_models` тАФ
  a new async helper in `providers.rs` that builds a default-registry client and
  calls `InferenceClient::list_models` тАФ off the UI task and feeds the result back
  as `Msg::Models`.
- `update` replaces the rows on a non-empty result; on error or an empty catalog it
  **keeps the static catalog** and notes why. The free-text path (Enter on a typed
  id the filter doesn't match) still lets any model id through, mirroring the CLI's
  unchecked `/model <id>`.

### Session switcher (`Effect::ListSessions`/`ResumeSession` тЖТ `Msg::SessionList`/`SessionResumed`)

- `Overlay::Sessions(SessionSwitcher)` wraps a `Picker<SessionSummary>`.
- A new `chat_session_summaries` helper reads the session store (blocking, run on
  `spawn_blocking` like the cloud fetches) and returns id/source/message-count rows,
  newest first.
- Selecting a row fires `Effect::ResumeSession(id)`, which replays the session via
  the existing `load_chat_session_state` and returns the rebuilt `SessionState`.
  `update` swaps in the state, resets the conversation/turn, refreshes usage, and
  posts a "ResumedтАж" notice тАФ the in-TUI analogue of the CLI resume path.

### Usage meter

- A `SessionUsage { estimated_tokens, messages }` on `App`, recomputed from the
  session transcript after each turn and after a resume. The harness does not
  propagate provider `Usage` counters through `TurnOutcome`, so this is a
  content-length estimate (~4 chars/token) rendered with a leading `~` and a `k`
  abbreviation so it fits the status bar.

## Tests

Pure `update` + `ratatui::backend::TestBackend`, following the existing TUI style:

- F2 opens the picker and fires `FetchModels`; `Msg::Models` replaces the rows; an
  empty result falls back to the catalog; free-text Enter sets any model id.
- `/switch` opens the switcher and fires `ListSessions`; `Msg::SessionList`
  populates it and selecting a row fires `ResumeSession`; `Msg::SessionResumed`
  resets the conversation and refreshes usage.
- The status bar renders the usage meter (count + `~N tok`, with `k` abbreviation),
  and the session switcher overlay renders its rows.

## Out of scope (follow-ups)

Writable org switching (needs a Clerk token re-mint flow), mouse support /
configurable themes, and exact provider-reported token usage (needs the harness to
thread `Usage` through `TurnOutcome`).
